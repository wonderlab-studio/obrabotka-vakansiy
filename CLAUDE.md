# Обработка вакансий — сервис-помощник в поиске работы

## Идея проекта

Веб-сервис, который помогает соискателю быстрее и качественнее откликаться на вакансии.
Пользователь даёт ссылку на вакансию (для начала — только hh.ru), сервис:

1. **Извлекает из вакансии** структурированные данные:
   - требования к кандидату;
   - информация о компании;
   - задачи для будущего сотрудника;
   - бонусы/плюшки для сотрудников.
2. **Сопоставляет опыт пользователя с требованиями.** Пользователь предоставляет тезисы
   о своём опыте (свободный список), сервис:
   - выбирает тезисы, которые релевантны требованиям вакансии;
   - выписывает требования, под которые у пользователя нет подходящего опыта (пробелы).
3. **Генерирует сопроводительное письмо** из:
   - приветствия;
   - мотивации работать именно в этой компании (на основе информации о компании, задачах
     и бонусах);
   - выбранных тезисов об опыте, релевантных вакансии;
   - заключения.

## Формат

Всё реализуется как **веб-интерфейс в одну страницу (SPA/одностраничник)** с аккуратным
дизайном и удобным UX: вставил ссылку → увидел разбор вакансии и сопоставление опыта →
получил готовое сопроводительное письмо. Результат **не редактируется прямо в сервисе** —
только просмотр/копирование; правки при необходимости делаются пользователем вне сервиса.

## Технологии и архитектура (решено)

- **Извлечение данных с hh.ru** — HTML-парсинг страницы вакансии (без официального API).
- **Фронтенд** — основная часть интерфейса.
- **Бэкенд** — минимальный, нужен как минимум для хранения API-ключа LLM (чтобы не светить
  его на клиенте); вероятно также удобен для самого HTML-парсинга hh.ru (обход CORS/
  антибот-защиты) — уточняется.
- **LLM** — доступ через API LLM-провайдера (конкретный провайдер — уточняется).
- **Хранение тезисов опыта пользователя** — сессионно, без персистентной БД пользовательских
  данных.

- **Авторизация** — не нужна, сервис со свободным доступом без аккаунтов.

## UI (одна страница, сверху вниз)

1. Поле ссылки на вакансию hh.ru + кнопка «Обработать» → бэкенд парсит HTML вакансии →
   LLM структурирует описание на 4 блока.
2. Блок «О компании».
3. Блок «Требования» (список, по пункту на требование).
4. Блок «Задачи».
5. Блок «Бонусы».
6. Поле «Тезисы опыта» (многострочное, по тезису на строку, с подсказкой по формату).
7. Поле «Контакты» (многострочное).
8. Кнопка «Сгенерировать сопроводительное» → один LLM-запрос сопоставляет требования и
   тезисы (подсветка требований зелёным/красным) и генерирует письмо.
9. Блок «Сопроводительное письмо» + кнопка копирования в буфер.

Также в макете (см. ниже) есть кнопка «↺ сбросить» в шапке и счётчик совпадений «N/M ✓»
рядом с легендой блока «Требования» — не было в изначальной разметке, взять в реализацию.

Подробности и промпты для LLM — в `requirements.md`, разделы 4.1, 4.2, 5.5.

вёрстки/стилей/адаптива — не финальный прод-код, содержит mock-данные и упрощённые

`requirements.md` и принятые по ним решения — см. `requirements.md`, раздел 4.2.

## Статус

Реализован и работает локально, проверено пользователем на реальной вакансии hh.ru — обе
кнопки («Обработать» и «Сгенерировать сопроводительное») отрабатывают на практике. Стек:
Node.js/Express-бэкенд (`server/`) + статический фронтенд на ванильном JS
(`public/index.html`, адаптирован из дизайн-макета) + Google Gemini API для обоих промптов
из `requirements.md` (5.5.1, 5.5.2), с фолбэком на локальный Ollama при исчерпании квоты —
см. `requirements.md`, раздел 5.6. Запуск: `npm install`, ключ в `.env` (см. `.env.example`),
`npm start` → `http://localhost:3000`.

Проверено реальными вызовами: оба промпта (структурирование вакансии, сопоставление +
генерация письма) отдают корректный JSON по согласованной схеме, включая фолбэк-цепочку
`gemini-2.5-flash` → `gemini-2.5-flash-lite` → локальный Ollama (у пользователя уже был
запущен Ollama с моделью `gemma-4-E4B-it`). Из найденного при тестировании: Gemini не всегда
возвращает `requirement` в `requirements_status` дословно (перефразирует), поэтому
сопоставление статусов с требованиями на фронтенде сделано по порядку элементов массива, а не
по точному совпадению текста; локальная Ollama-модель, наоборот, вернула требования дословно.

По жалобе пользователя на медленную работу и низкую точность письма внесены правки в
`server/gemini.js` (см. `requirements.md`, разделы 5.5.2, 5.6): убраны внутренние повторы
вызова одной и той же модели Gemini (до 3 попыток с задержкой 1с/2с/4с — тратили до ~14с
впустую перед переходом к следующему провайдеру), добавлены таймауты на вызов (20с Gemini,
60с Ollama) и кэш «модель временно исчерпана» на 6 часов после 429, чтобы не тратить время на
заведомо обречённый повторный запрос к той же модели в тот же день. Из промпта 5.5.2 убрана
фраза-«готовность осваивать невыполненные требования» — LLM либо не добавляла её, либо в
погоне за ней пропускала часть подходящих тезисов; теперь `experience_part` — только
перечисление подошедших тезисов, без изменения формулировок, без пропусков. Также обнаружилось,
что Gemini слегка перефразирует текст тезисов даже при явном запрете в промпте — исправлено
структурно: LLM теперь возвращает только номера подошедших тезисов (`matched_thesis_indices`),
а текст для письма бэкенд берёт дословно из исходного массива тезисов.

Цепочка фолбэков расширена четвёртым звеном: `gemini-2.5-flash` → `gemini-2.5-flash-lite` →
облачная **Ollama** (`https://ollama.com`, ключ `OLLAMA_CLOUD_API_KEY`) → локальная Ollama —
добавлено по просьбе пользователя, готового завести платный ключ Ollama Cloud, чтобы резерв
был быстрее локальной CPU-генерации, но по-прежнему бесплатный локальный Ollama остаётся
последним звеном на случай недоступности всего облака.

По жалобе пользователя на нестрогий подбор тезисов (LLM включала в письмо тезис про
прототипирование в Figma/Axure для вакансии, где такого требования не было) в промпт 5.5.2
добавлено явное лексическое правило соответствия: требование и тезис считаются совпадающими
только при общем существительном/глаголе/аббревиатуре, называющих конкретный навык/
технологию/артефакт (без учёта общих слов вроде «проект», «работа»), включая межъязыковые
аббревиатуры одного артефакта (ФТ/FR). Схема ответа LLM не менялась (пользователь явно
попросил не усложнять её ради сохранения скорости ответа) — см. `requirements.md`, 5.5.2.

Задеплоен в интернет: код запушен в публичный GitHub-репозиторий
(`https://github.com/wonderlab-studio/obrabotka-vakansiy`), сервис развёрнут на Railway из
этого репозитория (автодеплой из ветки `master`, переменные окружения заданы вручную в
Railway Variables по `.env.example`, `PORT` берёт из `process.env.PORT` — см. `server/index.js`).
Пользователь проверил обе кнопки на реальной вакансии hh.ru на проде — работают, парсинг
страницы вакансии с IP Railway проходит успешно.

Обнаружено на проде: кнопка «Сгенерировать сопроводительное» один раз потребовала повторного
нажатия — оба Gemini (основная и lite) и облачная/локальная Ollama были заняты/превысили таймаут
одновременно, вся фолбэк-цепочка не успела отработать с первого раза.

Цепочка расширена и переработана (2026-07-09, см. `requirements.md` 5.6): вместо
Gemini/Gemini-lite/Ollama теперь 7 бесплатных облачных провайдеров в три уровня + платная
облачная Ollama последним резервом — Groq и Cerebras гоняются параллельно (первый успешный
ответ), резервы — Gemini, затем Mistral (резерв Cohere), затем NVIDIA NIM (резерв GitHub
Models). Локальная Ollama отключена от цепочки (была самым медленным звеном, стала избыточной
при 7 облачных резервах). Код — `server/llm.js` (переименован из `gemini.js`). Живая проверка
новых 6 провайдеров (Groq, Cerebras, Mistral, Cohere, NVIDIA NIM, GitHub Models) на реальных
запросах — ожидает, пока пользователь заведёт для них API-ключи (см. `.env.example`); пока
проверен только сценарий, где все новые ключи отсутствуют и цепочка корректно проваливается до
уже настроенного резерва Gemini.

По серии диагностических экспериментов с промптом 5.5.2 (2026-07-13, вне боевой схемы —
временный промпт с доп. полем `match_explanations`, который к каждому подобранному тезису
приписывал требование и ключевое слово совпадения, в прод не попал) в правило соответствия
внесены два уточнения: в список общих слов, не считающихся совпадением, добавлены «бизнес»,
«аналитика», «разработка» (модель цеплялась за них как за ложные признаки релевантности);
отдельно добавлено числовое правило для требований о стаже вида «опыт от N лет» — тезис
засчитывается при указанном стаже ≥ N, даже если название должности не совпадает дословно. Оба
уточнения проверены на Gemini вживую и работают корректно. При этом первая живая проверка
уровня A цепочки вскрыла два бага: у Cerebras была нерабочая модель по умолчанию
(`llama-3.3-70b` не существует для реального ключа — исправлено на `gpt-oss-120b`, реальный
список моделей получен через `/v1/models`); и Groq заметно хуже Gemini соблюдает строгое
лексическое правило (подбирает тезисы почти без разбора и одновременно пропускает очевидные
совпадения) — из-за сломанной Cerebras весь уровень A на практике работал через один Groq, это
и объясняет замеченные пользователем на проде странные подборки тезисов. См. подробности и
возможные следующие шаги в `requirements.md`, раздел 5.6.

Ожидается: подтверждение решения по формату company/tasks/bonuses (раздел 4.2 requirements.md),
решение по кнопке «сгенерировать заново», решение по замене NVIDIA NIM на другого провайдера
(пользователь не смог получить ключ; рекомендация — OpenRouter, ожидает подтверждения), решение
по дальнейшей судьбе Groq в уровне A (если низкая строгость сопоставления будет повторяться),
заведение оставшихся API-ключей и живая проверка расширенной цепочки на проде.

## Документы проекта

- `requirements.md` — рабочие требования и уточнения по проекту (не в git, личный рабочий
  документ, см. `.gitignore`).
- `skills/develop/skill.md` — скилл для разработки в рамках этого проекта.
- `/дизайн/cover-letter-ui.html` — дизайн-макет интерфейса (референс для вёрстки).
- `server/` — бэкенд (Express): `index.js` (роуты), `hhParser.js` (парсинг hh.ru),
  `llm.js` (промпты + вызовы LLM с фолбэк-цепочкой из 7 провайдеров + облачная Ollama,
  см. `requirements.md` 5.6).
- `public/index.html` — фронтенд (статика, отдаётся бэкендом).
- `.env` / `.env.example` — ключи LLM-провайдеров и настройки (`.env` не в git и не читается
  Claude Code, см. `.gitignore` / `.claudeignore`).
