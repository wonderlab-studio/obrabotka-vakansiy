# Разработка в проекте "Обработка вакансий"

## Контекст

Перед началом работы прочитай `CLAUDE.md` в корне проекта — там описана идея сервиса,
и `requirements.md` — там актуальные требования и уточнения от пользователя.
Если `requirements.md` пуст или неполон, а задача требует детали, которых там нет —
уточни у пользователя, не додумывай архитектуру самостоятельно.

## Как подходить к разработке

- Проект — одностраничный веб-сервис. Не разрастайся на лишние страницы/роуты без запроса.
- Три основных этапа пайплайна держать разделёнными и тестируемыми по отдельности:
  1. извлечение данных с hh.ru (требования, о компании, задачи, бонусы);
  2. сопоставление тезисов опыта пользователя с требованиями (найденные соответствия +
     пробелы);
  3. генерация сопроводительного письма.
- Дизайн и UX важны: сервис должен быть приятным и удобным, а не просто рабочим прототипом.
- Результат (сопоставление опыта, сопроводительное письмо) **не редактируется прямо в
  интерфейсе** — только просмотр/копирование. Не добавляй инлайн-редактирование результата,
  если явно не попросили пересмотреть это решение.
- Не добавляй поддержку других источников вакансий кроме hh.ru, пока это не попросили явно.
- Извлечение данных с hh.ru — HTML-парсинг страницы вакансии (официальный API hh.ru не
  используется).
- Тезисы опыта пользователя хранятся сессионно, не персистентно — не добавляй БД для их
  хранения без явного запроса.
- API-ключ LLM не должен попадать на клиент — операции, требующие ключа, идут через бэкенд.
- Перед реализацией крупного куска функциональности сверяйся с `requirements.md` —
  он источник истины по деталям, которые дополняет пользователь.

## После изменений

- Проверяй фактическую работу фичи через браузер/запуск сервиса, а не только по коду —
  особенно для UI и для этапа извлечения данных с реальной страницы hh.ru.
